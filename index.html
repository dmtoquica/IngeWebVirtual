Modelos de datos: lo mínimo necesario

(10 min)

Un modelo de datos no es aún una base de datos. Es simplemente cómo representamos la información.

Ejemplo de modelo:

{
  "id": 1,
  "title": "Preparar entrega Ingeniería Web",
  "completed": false
}

¿Por qué es importante hoy?

  • 1. Las pruebas dependen del modelo.
  • 2. El contrato depende del modelo.
  • 3. La cobertura depende del comportamiento esperado.

📌 Diseño primero, implementación después.

5

Contratos de servicio: el acuerdo

(10 min)

El contrato define:

📥 Qué recibe 📤 Qué devuelve ⚠️ En qué casos falla

Ejemplo Contractual: POST /tasks

Request

{
  "title": "Hacer pruebas"
}

Response (201)

{
  "id": 2,
  "title": "Hacer pruebas",
  "completed": false
}

Error (400)

{
  "error": "Title is required"
}

📌 Mensaje clave:

El contrato es lo que permite que frontend, pruebas y CI/CD trabajen sin ambigüedades.

6

Relación directa con pruebas y CI/CD

(10 min)

¿Por qué diseñamos así el backend? Porque luego todo esto se automatiza.

Pruebas unitarias

  • • Prueban funciones del Service
  • • No dependen de HTTP
  • • Son rápidas y confiables

Integración

  • • Validan endpoints
  • • Verifican contratos
  • • Detectan rupturas tempranas

Pipeline CI/CD

  • • Ejecuta pruebas automáticamente
  • • Mide cobertura
  • • Falla si baja la calidad

"El pipeline valida decisiones de diseño, no solo líneas de código."
